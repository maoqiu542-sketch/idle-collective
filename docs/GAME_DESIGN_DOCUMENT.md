# Idle Collective 游戏策划案

> 基于100个游戏设计原则框架分析

---

## 一、游戏概述

### 1.1 基本信息

| 项目 | 内容 |
|------|------|
| **游戏名称** | Idle Collective |
| **游戏类型** | 放置模拟 + 策略管理 |
| **核心灵感** | RimWorld 工作优先级系统 |
| **目标平台** | PC (Electron桌面应用) |
| **目标受众** | 放置游戏爱好者、模拟经营玩家 |

### 1.2 核心概念

**一句话描述**: 管理一群有自主意识的角色，在自然生成的世界中采集资源、建造家园、满足需求。

## 二、核心游戏循环

### 2.1 主循环设计

```
┌─────────────────────────────────────────────────────────────┐
│                      核心游戏循环                             │
│                                                              │
│   角色生成需求 → AI评估任务 → 执行行为 → 满足需求/获得资源    │
│        ↑                                          │         │
│        └──────────────────────────────────────────┘         │
│                                                              │
│   外层循环：建造建筑 → 提升效率 → 解锁新内容                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 循环评估 (基于设计原则)

| 设计要求 | 评估 | 说明 |
|----------|------|------|
| 易于理解 | ✅ | 需求条清晰，任务目标明确 |
| 易于操作 | ✅ | 无需手动控制，AI自动执行 |
| 有趣 | ⚠️ | 需要增加决策点和随机事件 |
| 直接反馈 | ✅ | 资源变化、状态变化即时可见 |
| 灵活 | ✅ | 可调整工作优先级 |
| 可扩展 | ✅ | 可添加新任务类型、新需求 |
| 可组合 | ✅ | 多角色协作完成任务 |
| 可演化 | ⚠️ | 需要添加长期成长系统 |

## 三、系统设计

### 3.1 角色系统

#### 需求系统 (Need System)

基于 **马斯洛需求层次理论** 设计：

| 需求层次 | 需求类型 | 游戏机制 | 紧急权重 |
|----------|----------|----------|----------|
| 生理需求 | 饥饿、休息 | 需要进食和睡眠 | 2.0-2.5 |
| 安全需求 | 安全感 | 避免危险区域 | 2.5 |
| 社交需求 | 社交 | 与其他角色互动 | 0.4 |
| 尊重需求 | 舒适、娱乐 | 良好的生活环境 | 0.3-0.5 |

**设计原则应用**:
- **心流通道**: 需求衰减速度与资源获取速度平衡
- **渐进式揭示**: 初始只显示基础需求，随游戏进展解锁高级需求

#### 职业系统

| 职业 | 核心技能 | 效率加成 | 设计定位 |
|------|----------|----------|----------|
| 采集者 | 采集、挖矿 | +20% | 资源获取专家 |
| 建造者 | 建造 | +30% | 基础设施建设者 |
| 农夫 | 烹饪 | +20% | 食物生产者 |
| 战士 | 战斗 | +50% | 安全保障者 |

**设计原则应用**:
- **有意义的选择**: 玩家需要根据当前需求选择招募职业
- **涌现玩法**: 不同职业组合产生不同的聚落发展策略

### 3.2 AI系统

#### 效用评估系统 (Utility AI)

```
效用值 = 基础分 × 需求权重 × 距离系数 × 紧急系数 × 技能系数
```

**设计原则应用**:
- **反馈循环模式**: 高效用任务完成 → 获得奖励 → 提升技能 → 更高效用
- **风险回报模式**: 远距离任务效用降低，但可能获得稀有资源

#### 任务优先级体系

| 优先级 | 任务类型 | 示例 |
|--------|----------|------|
| 紧急 (1) | 生命威胁 | 灭火、救援、治疗 |
| 高 (2) | 需求满足 | 进食、睡眠 |
| 普通 (3) | 日常工作 | 采集、建造、种植 |
| 低 (4) | 休闲活动 | 娱乐、社交、冥想 |

**设计原则应用**:
- **动态难度调整**: 根据角色状态自动调整任务优先级
- **玩家代理**: 玩家可手动调整工作优先级设置

### 3.3 地图系统

#### 地形生成算法

使用 **分形噪声 (Fractal Noise)** 生成自然地形：

```
地形 = f(高度噪声, 湿度噪声, 细节噪声)
```

**设计原则应用**:
- **空间叙事**: 地形聚集形成自然区域特征
- **视觉玩家引导**: 资源聚集点引导玩家探索

#### 资源分布设计

| 资源 | 关联地形 | 聚集半径 | 丰富度范围 |
|------|----------|----------|------------|
| 木材 | 森林 | 3-7格 | 50-100% |
| 石材 | 山地 | 3-7格 | 50-100% |
| 食物 | 草地 | 3-7格 | 50-100% |

**设计原则应用**:
- **有意义的选择**: 玩家需要决定在哪个资源聚集点建立基地
- **节奏控制**: 资源耗尽后需要迁移，创造紧张感

## 四、玩家体验设计

### 4.1 新玩家体验

| 阶段 | 时间 | 内容 | 设计目标 |
|------|------|------|----------|
| 吸引 | 0-30秒 | 看到角色自动工作 | 展示核心玩法 |
| 引导 | 30秒-3分钟 | 第一个角色采集资源 | 建立基本认知 |
| 深入 | 3-10分钟 | 建造第一个建筑 | 体验成长感 |
| 留存 | 10分钟+ | 解锁新职业、新建筑 | 创造长期目标 |

**检查清单验证**:
- [x] 核心循环能在30秒内体验
- [x] 目标清晰易懂
- [x] 控制简单直观（无需手动控制）
- [x] 反馈即时
- [ ] 需要增加深度支持长期游玩

### 4.2 心流设计

```
挑战程度
    ↑
高  │         ╱──────╲
    │       ╱          ╲
    │     ╱              ╲
    │   ╱                  ╲
中  │ ╱                      ╲
    │╱                        ╲
低  └──────────────────────────→ 技能水平
    低                        高
    
    游戏时间 →
```

| 游戏阶段 | 挑战程度 | 技能需求 | 心流状态 |
|----------|----------|----------|----------|
| 初期 | 低 | 低 | 学习期 |
| 发展期 | 中 | 中 | 成长期 |
| 成熟期 | 高 | 高 | 掌控期 |
| 危机期 | 很高 | 高 | 紧张期 |

### 4.3 奖励系统

| 奖励类型 | 频率 | 示例 | 设计目标 |
|----------|------|------|----------|
| 即时奖励 | 持续 | 资源增加 | 维持参与感 |
| 短期奖励 | 分钟级 | 建筑完成 | 创造成就感 |
| 中期奖励 | 小时级 | 新角色加入 | 提供期待 |
| 长期奖励 | 天级 | 聚落升级 | 创造目标 |

## 五、技术架构

### 5.1 分层架构

```
┌─────────────────────────────────────────────────────────┐
│                    表现层 (Presentation)                 │
│  React Components, UI, Views                           │
├─────────────────────────────────────────────────────────┤
│                    应用层 (Application)                  │
│  Game Loop, Systems Coordinator, Event Bus             │
├─────────────────────────────────────────────────────────┤
│                    领域层 (Domain)                       │
│  Entities, Systems, AI, Rules                          │
├─────────────────────────────────────────────────────────┤
│                    数据层 (Data)                         │
│  Config, Save/Load, Resource Management                │
├─────────────────────────────────────────────────────────┤
│                    基础设施 (Infrastructure)             │
│  Electron, File System, Desktop Integration            │
└─────────────────────────────────────────────────────────┘
```

### 5.2 设计模式应用

| 设计模式 | 应用场景 | 实现方式 |
|----------|----------|----------|
| 发布订阅 | 系统间通信 | EventBus |
| 依赖注入 | 服务管理 | ServiceContainer |
| 状态模式 | 角色状态 | CharacterState |
| 策略模式 | AI决策 | UtilityEvaluator |
| 工厂模式 | 任务创建 | TaskFactory |

## 六、商业化设计

### 6.1 核心价值主张

**给玩家**: 无需频繁操作，观看角色自主生活的乐趣

**差异化**: 
- 相比传统放置游戏：更智能的AI，更丰富的角色个性
- 相比RimWorld：更轻松，更适合挂机

### 6.2 潜在商业化点

| 功能 | 商业化方式 | 设计考量 |
|------|------------|----------|
| 角色外观 | 付费皮肤 | 不影响游戏平衡 |
| 建筑皮肤 | 付费装饰 | 纯视觉内容 |
| 加速道具 | 内购 | 需谨慎平衡 |
| 新地图主题 | DLC | 扩展内容 |

## 九、设计原则总结

### 已应用的设计原则

| 原则 | 应用位置 | 效果 |
|------|----------|------|
| **心流通道** | 需求衰减与资源获取平衡 | 维持玩家参与度 |
| **反馈循环** | 任务完成→奖励→技能提升 | 强化玩家行为 |
| **渐进式揭示** | 需求系统、建筑解锁 | 降低认知负担 |
| **有意义的选择** | 职业选择、基地选址 | 增加策略深度 |
| **涌现玩法** | 职业组合、资源分布 | 创造独特体验 |
| **玩家代理** | 工作优先级设置 | 增加控制感 |

### 待加强的设计原则

| 原则 | 建议应用 |
|------|----------|
| **节奏控制** | 添加危机事件，创造紧张-放松节奏 |
| **社交货币** | 添加分享功能，展示聚落成就 |
| **风险回报** | 添加危险区域，高回报资源 |
| **动态难度调整** | 根据聚落发展调整事件难度 |

## 十、结论

Idle Collective 是一款具有清晰核心循环和深度AI系统的放置模拟游戏。项目已完成核心框架搭建，具备：

**优势**:
- 完善的效用AI系统
- 自然的地形生成
- 清晰的需求驱动机制
- 模块化的技术架构

**待改进**:
- 增加长期目标和成长系统
- 丰富随机事件和危机系统
- 完善UI和视觉反馈
- 添加音效和音乐

**建议优先级**:
1. 完善资源采集→建造→升级的核心循环
2. 添加随机事件系统增加变数
3. 实现存档系统支持长期游玩
4. 优化UI和视觉反馈
