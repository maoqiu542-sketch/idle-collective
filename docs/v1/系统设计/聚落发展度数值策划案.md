## 📋 文档说明

本文档为 `Idle Collective` 的**聚落发展度数值策划案**，面向策划、程序、测试团队，描述发展度在 V1 中的系统职责、数值构成、公式设计、边界条件与调优方向。

**本文档重点是数值策划，不涉及具体程序实现方式。**

## 一、系统概述

### 1.1 系统定位

聚落发展度是 V1 的**阶段控制数值系统**。  
它用于衡量玩家当前聚落的整体成长阶段，并驱动以下系统：

- Boss 难度分层
- 招募质量上限
- 中期阶段推进判断

### 1.2 核心价值

| 价值维度 | 描述 |
|---------|------|
| **阶段价值** | 把聚落成长压缩成一个可读、可控的阶段数值 |
| **平衡价值** | 避免玩家只靠单一系统推进所有阶段 |
| **反馈价值** | 让研究、建设、Boss、战备能共同作用于阶段推进 |

### 1.3 设计目标

| 目标 | 具体要求 |
|-----|---------|
| **多线协同** | 研究、建筑、核心零件、战备都要有存在感 |
| **防止偏科** | 不允许单一维度直接刷满发展度 |
| **数值稳定** | 公式应对局部数值调整有一定容错，不轻易失真 |
| **易于调优** | 各输入项应便于单独调权重和校验 |

### 1.4 系统关联

**输入系统**：
- 科技系统
- 建筑系统
- Boss / 核心零件系统
- 战备与战力系统

**输出系统**：
- Boss 难度池
- 招募质量上限
- 阶段推进提示

## 二、功能需求

### 2.1 核心功能模块

#### 模块一：发展度构成

发展度由四个维度组成：

| 维度 | 含义 |
|-----|------|
| 研究 | 聚落科技层级 |
| 建筑 | 聚落基础成熟度 |
| 核心零件 | Boss 突破投入 |
| 战备 | 当前可投入 Boss 的准备程度 |

#### 模块二：阶段映射

发展度主要用于：

| 输出项 | 作用 |
|------|------|
| Boss 难度 | 决定当前刷新的 Boss 池 |
| 招募质量上限 | 控制候选成员成长质量 |
| 阶段提示 | 告诉玩家当前聚落进入了哪个阶段 |

#### 模块三：偏科惩罚

系统需要轻度惩罚“只堆一条线”的行为。  
目标不是硬锁玩家，而是让研究、建筑、核心零件三条线都持续有效。

## 三、旧公式问题分析

### 3.1 当前旧公式

```text
development = clamp(
  key_research_count * 20 +
  key_building_level_sum * 8 +
  core_part_spent * 2 +
  active_combat_power / 50,
  0,
  100
)
```

### 3.2 存在的问题

| 问题 | 说明 |
|-----|------|
| 建筑权重过高 | `key_building_level_sum * 8` 单项理论上限已超过 100 |
| 研究跳变过硬 | `key_research_count` 档位少，但每档跃迁过大 |
| 战力波动太直接 | 原始战力直接进公式，容易放大战备或装备波动 |
| 缺少协同约束 | 研究、建筑、核心零件之间没有均衡激励 |
| 抗调参能力弱 | 某个系统微调后，发展度整体可能被拉歪 |

## 四、推荐数值方案

### 4.1 设计思路

- 所有输入先标准化到 `0-1`
- 研究、建筑、核心零件作为主权重
- 战备作为辅助权重
- 战力使用 `sqrt` 压缩
- 加入轻度偏科惩罚

### 4.2 推荐公式

```text
research_score = clamp(completed_key_research / 3, 0, 1)

building_score = clamp(
  0.15 * clamp(house_level / 3, 0, 1) +
  0.10 * clamp(kitchen_level / 2, 0, 1) +
  0.10 * clamp(farm_level / 3, 0, 1) +
  0.10 * clamp((lumbermill_level + quarry_level) / 6, 0, 1) +
  0.20 * clamp(research_bench_level / 3, 0, 1) +
  0.10 * clamp(warehouse_level / 2, 0, 1) +
  0.10 * clamp(recruit_station_level / 2, 0, 1) +
  0.10 * clamp(barracks_level / 2, 0, 1) +
  0.05 * clamp(trade_station_level / 2, 0, 1),
  0,
  1
)

core_score = clamp(core_part_spent / 12, 0, 1)

combat_score = clamp(sqrt(active_combat_power / target_combat_power), 0, 1)

base_development =
  100 * (
    0.35 * research_score +
    0.30 * building_score +
    0.20 * core_score +
    0.15 * combat_score
  )

balance_factor = 0.85 + 0.15 * min(research_score, building_score, core_score)

development = floor(clamp(base_development * balance_factor, 0, 100))
```

### 4.3 权重说明

| 变量 | 权重 | 设计理由 |
|------|------|----------|
| `research_score` | 0.35 | 科技是阶段推进主线之一 |
| `building_score` | 0.30 | 建筑代表聚落成熟度，但不能一项刷满 |
| `core_score` | 0.20 | Boss 突破必须有存在感 |
| `combat_score` | 0.15 | 战备重要，但只能做辅助项 |

### 4.4 为什么不把金币放进发展度

- 金币是中介货币，不是阶段本身
- 若金币直接影响发展度，会把贸易站变成“拉 Boss 难度的建筑”
- 更合理的方式是仅让 `trade_station_level` 进入建筑成熟度，而不是让金币存量直接进公式

## 五、关键数值与建议范围

| 数值 | 建议范围 | 设计理由 |
|------|----------|----------|
| `completed_key_research` | 0 - 3 | 对应 V1 三条关键研究 |
| `core_part_spent` | 0 - 12 | 便于分阶段体现 Boss 投入 |
| `target_combat_power` | 500 - 800 | 定义 V1 后段合格战备线 |
| `balance_factor` | 0.85 - 1.00 | 轻度惩罚偏科，避免单线越级 |

## 六、边界条件与异常场景

| 场景 | 预期行为 |
|------|----------|
| 玩家只堆建筑，不做研究 | 建筑分仍上升，但无法单线刷满发展度 |
| 玩家只打 Boss，不做建设 | 核心零件分有限，不会直接撑爆发展度 |
| 玩家战力短时暴涨 | 战备分经压缩后只做辅助贡献 |
| 玩家金币暴增 | 不直接提高发展度，只提升后续选择能力 |
| 玩家没有贸易站 | 不阻断前期阶段理解，只影响中期金币节奏 |

## 七、测试与调优指标

### 7.1 建议 KPI

| KPI | 建议目标 |
|-----|----------|
| 发展度来源占比 | 研究/建筑/核心零件三项长期都要有效 |
| 同发展度段 Boss 胜率 | 40% - 70% |
| 单项来源占比上限 | 任一单项不应长期高于总分 50% |

### 7.2 验收标准

- [ ] 发展度不会被单项建筑等级直接刷满
- [ ] 研究、建筑、核心零件三条线都对发展度有显著贡献
- [ ] 战备影响存在，但不会压过主成长线
- [ ] Trade 加入后不会直接拉爆 Boss 难度
- [ ] 发展度阶段变化能被玩家清晰感知

---

## 八、结论

聚落发展度在 V1 中必须继续承担**阶段控制系统**职责。  
因此它的公式必须从“原始变量线性相加”改成“标准化加权 + 轻度偏科惩罚”的形式，才能保证多线成长都有效、Boss 难度可控、Trade 不会错误抢占阶段推进地位。
