# V1 架构文档

## 1. 架构目标

V1 的架构目标是建立清晰的单一事实来源，保证规划、自动执行、资源结算、招募、科技、Boss 和存档都能围绕同一套状态运行。

## 2. 分层结构

### 2.1 配置层

配置层负责静态数据：

- 建筑定义
- 职业定义
- 资源定义
- 科技定义
- Boss 定义
- 装备定义
- 全局策略预设定义

配置层不持有运行时状态。

### 2.2 领域层

领域层负责运行时规则：

- 角色
- 需求
- 资源
- 建筑
- 招募站
- 科技
- Boss
- 装备
- 聚落宜居度
- 聚落发展度

领域层是所有计算的权威来源。

### 2.3 调度层

调度层负责周期性执行：

- 需求衰减与恢复
- 优先级分配
- 生产建筑结算
- 商队刷新
- Boss 刷新
- 存档触发

### 2.4 表现层

表现层负责：

- 面板显示
- 按钮交互
- 提示与引导
- 战斗演出
- 状态反馈

表现层不直接修改核心状态，只能通过领域接口发起请求。

## 3. 单一事实来源规则

以下状态必须由领域层统一管理：

- 角色基础属性
- 角色需求
- 角色优先级
- 全局策略模式
- 资源数量
- 建筑等级与工作状态
- 招募站候选列表
- 科技进度
- Boss 刷新状态
- 装备库存
- 聚落宜居度
- 聚落发展度

UI 只读这些状态，不自行生成独立副本。

## 4. 核心状态契约

### 4.1 全局策略

```ts
globalStrategyPreset: 'none' | 'survival' | 'expand' | 'research' | 'boss'
priorityMode: 'preset' | 'manual'
```

### 4.2 聚落指标

```ts
settlementLivability: number // 0-100
settlementDevelopment: number // 0-100
```

### 4.3 资源与招募

```ts
coreParts: number
recruitmentStationState: {
  candidates: CharacterCandidate[]
  nextRefreshAt: number
  manualRefreshCost: ResourceCost
}
```

## 5. 存档范围

V1 存档至少保存：

- 当前资源
- 角色列表
- 角色优先级
- 全局策略
- 建筑状态
- 科技进度
- Boss 状态
- 装备库存
- 聚落宜居度
- 聚落发展度
- 招募站状态

## 6. 关键设计原则

- 规则必须可解释
- 计算必须可复现
- UI 必须只呈现结果，不持有真相
- 预设策略可以覆盖角色级优先级
- 关闭预设后，角色级优先级恢复可编辑

## 7. 验收标准

- 任何关键状态都能从领域层追溯到唯一来源。
- 预设策略与角色级优先级互斥逻辑稳定。
- 存档与读档后状态一致。
- UI 不会因为本地缓存造成状态漂移。
