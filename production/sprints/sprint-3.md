# Sprint 3 - V1 Demo 验收收口与 Trade MVP 稳定化

## 目标

让 `idle-collective` 当前版本达到可连续试玩、可解释、可验收的 `V1 Demo` 状态，并把 `Trade MVP` 收口到稳定可验证的最小版。

本轮不扩内容层，不开启 `V1.1` 新系统；只做主循环收口、经济校准、反馈统一、性能止血和回归验证。

里程碑对齐：以 `production/milestones/v1-demo.md` 的退出条件为准，以 `Sprint 2` 的 Trade/经济/发展度成果为输入，补齐人工试玩和阻塞问题闭环。

## 时间范围

- 开始：2026-04-06
- 结束：2026-04-19
- 总时长：14 日历天，按 10 个工作日估算
- Buffer：2 个工作日，专门留给试玩中暴露的新 P0/P1
- 计划可用容量：8 个工作日

## 本轮交付

- 完整人工试玩记录与问题单
- Trade MVP 稳定版（盈余交易、保留库存、开关、最近结果、收益展示）
- 金币经济闭环校准
- 发展度公式与反馈对齐
- 试玩阻塞项清理
- 存档/读档一致性验证
- Demo checklist 更新

## 验收标准

### 自动化测试

- `npm run typecheck` 通过
- `npm run test:run` 通过
- `npm run build` 通过
- `npm run package` 通过

### 人工验证场景

1. 新档启动到主界面，完成一次基础开局
2. 完成"规划 -> 资源增长 -> 建筑/研究 -> 招募 -> Boss -> 升级"的主循环体验
3. 建造并启用贸易站，验证盈余交易、保留库存、开关、最近成交结果
4. 验证金币从 Trade 流向招募、刷新、装备，而不是长期空转堆积
5. 收起底部面板后反复拖动地图，确认不卡到影响操作判断
6. 存档后读档，验证 Trade、发展度、招募状态、金币状态一致

### 退出条件

- 至少完成 1 次完整 30-45 分钟试玩并沉淀结果
- 无未分配负责人的 P0
- P1 若保留，必须有修复时间点或明确降级理由
- Demo checklist 完整更新，已知问题可追踪
- 满足 V1 Demo 里程碑退出条件中的启动、主循环、存档、可理解性要求

## 接口与产物

### 不新增

- 不新增新的独立玩法系统
- 不新增新的长期公开玩法接口

### 允许收紧

- `tradeState`：必须保证开关、保留库存、最近结果、估算收益在 UI 与存档中语义一致
- `developmentBreakdown`：四项来源与展示口径保持一致，不允许 UI 与公式分叉
- 存档快照：如需补字段，只允许做向后兼容扩展，不做破坏性 schema 变更

### 产物更新

- `production/sprints/sprint-3.md`
- `production/qa/demo-checklist.md`
- 必要时补充一份本轮试玩问题单或状态报告

## 风险

1. Trade 收益过强，压过建造、科技、Boss 的节奏
2. 金币 sink 仍然偏弱，导致中期经济失控
3. 地图拖动卡顿若未解决，会干扰对游戏节奏本身的判断
4. 读档一致性若存在问题，会直接降低试玩可信度
5. 若 Sprint 3 内出现新的 P0，应优先占用 buffer，自动下调 Should Have

## 假设

- 默认本轮主目标选用"验收收口"，不是"性能优先"或"均衡推进"
- 默认按 2 周 Sprint 执行，起始时间接 Sprint 2 之后
- 默认 Trade 继续保持 MVP 边界：不做商人、市场波动、多货币、运输链、装饰扩展
- 默认地图拖动卡顿是本轮必须处理的试玩阻塞项之一，但只要求降到"不影响试玩判断"，不要求一次性做完全部性能重构
- 默认本轮不引入新的破坏性存档结构调整；若必须扩字段，按兼容旧档处理

## 备注

- 本轮核心是"收口"而非"扩展"，所有工作服务于 V1 Demo 可验收状态
- 若试玩暴露重大设计问题，优先记录并评估是否影响 V1 基线
