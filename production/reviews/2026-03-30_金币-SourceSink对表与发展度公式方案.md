# 金币 Source/Sink 对表与发展度公式重权重方案

日期：2026-03-30

## 1. 目标与原则

- 金币在 V1 中应被明确定位为“中期规划货币”，主要服务 `招募`、`刷新`、`装备` 三类决策。
- 金币 faucet 应尽量单一、可预测、自动化，避免把系统做成多入口掉币。
- 金币 sink 应优先服务现有主循环，而不是把金币变成建筑升级主资源。
- 发展度应继续承担“阶段控制”职责，但不应被单一系统刷满，也不应直接被金币存量驱动。

## 2. 金币 Faucets 对表

| 项目 | 类型 | 当前定位 | 阶段 | 优先级 | 说明 |
|------|------|----------|------|--------|------|
| 贸易站自动交易盈余木材/石材/食物 | 常驻 faucet | 主 faucet | 中期开始 | P0 | 当前设计里唯一明确、稳定、可持续的金币来源，应保持为 V1 的正式主入口 |
| 贸易站等级提升带来的效率增益 | faucet 放大器 | 次级放大器 | 中后期 | P1 | 不直接产金币，但会提升单位时间金币流速 |
| 资源价格差异 | faucet 调节器 | 结构性调节 | 全程 | P1 | 木/石/食物价格用于控制哪类生产更适合转币 |

### Faucet 结论

- V1 不建议新增第二个常驻金币 faucet。
- Boss 不建议掉金币，Boss 奖励继续保持为核心零件。
- 研究不建议直接产金币，研究应通过提升生产、交易效率或招募质量间接增强经济。
- 金币存量不应反向产生更多金币，避免形成滚雪球失控。

## 3. 金币 Sinks 对表

| 项目 | 类型 | 当前定位 | 阶段 | 优先级 | 说明 |
|------|------|----------|------|--------|------|
| 招募候选成员 | 核心 sink | 主 sink | 中期开始 | P0 | 最核心的金币去向，直接把资源规划转成劳动力与生产力 |
| 手动刷新招募站 | 弹性 sink | 压力 sink | 中期开始 | P0 | 解决“有钱但刷不到人”的主动消费需求，同时给金币提供稳定消耗口 |
| 装备抽取或购买 | 弹性 sink | 中后期 sink | 中后期 | P1 | 装备是辅助成长系统，适合承担中后期金币蓄水池功能 |
| 贸易站建造成本中的金币部分 | 一次性 sink | 解锁 sink | 中期切入点 | P0 | 用于约束贸易站不能成为早期开局最优解 |
| 贸易站升级中的金币成本 | 可选 sink | 弱 sink | 中后期 | P2 | 若加入，建议只作为辅助成本，不能抢走基础资源与核心零件的地位 |

### Sink 结论

- 招募购买必须是最强、最稳定、最有满足感的金币 sink。
- 刷新费用必须存在，否则金币在“候选质量差”时缺少主动消耗口。
- 装备消费适合做中后期弹性 sink，但权重应低于招募。
- 建筑升级不应普遍金币化，否则会冲掉基础资源与核心零件的定位。

## 4. 优先级建议

| 优先级 | 项目 | 目标 |
|--------|------|------|
| P0 | 贸易站主 faucet 跑通 | 让木/石/食物溢出能稳定转为金币 |
| P0 | 招募购买价格曲线 | 确保金币有持续且有价值的主要出口 |
| P0 | 招募刷新价格曲线 | 确保金币不会在候选池不理想时堆死 |
| P0 | 发展度公式重写 | 避免单一系统刷满发展度，恢复阶段控制能力 |
| P1 | 装备金币 sink | 给中后期金币提供可选但有吸引力的消耗口 |
| P1 | 贸易站 UI 与反馈 | 让玩家看懂“为什么卖、卖了什么、赚了多少” |
| P2 | 金币溢出兜底机制 | 仅在装备系统未就绪时补充临时压力阀 |

## 5. 需要定义的关键数值

| 数值 | 建议范围 | 设计理由 |
|------|----------|----------|
| `trade_cycle_sec` | 15-30 秒 | 交易反馈不能太慢，也不能快到像瞬时兑换 |
| `wood_price` | 1.0-1.2 | 木材既是常用建材又是交易资源，不宜过高 |
| `stone_price` | 1.0-1.2 | 石材同理，避免石材生产直接优于其他规划 |
| `food_price` | 0.4-0.6 | 食物承担生存职责，必须天然更不适合卖 |
| `reserve_stock_default` | 木 120 / 石 80 / 食 100 | 与现有贸易文档一致，适合作为默认线 |
| `pending_buffer_ratio` | 100% 队列成本预留 | 防止“刚攒够建材又被系统卖掉” |
| `max_trade_per_cycle` | 20 / 35 / 50 | 与现有贸易站 1-3 级建议一致，抑制前期瞬时爆币 |
| `trade_efficiency` | 1.00 / 1.15 / 1.30 | 让升级增强体验，但不重写资源层级关系 |
| `recruit_cost_per_unit` | 约等于 8-12 分钟正常贸易收入 | 招募应是需要规划但不会遥不可及的主 sink |
| `refresh_cost` | 约等于 2-4 分钟正常贸易收入 | 刷新应可常用但不能无脑连点 |
| `refresh_cost_growth` | 每次连续刷新 +25% 到 +40% | 限制赌池行为，保护招募池波动价值 |
| `equipment_cost_per_unit` | 约等于 6-10 分钟正常贸易收入 | 装备应是可选消费，不应压过招募 |
| `gold_overflow_alert_line` | 大于下一个核心 sink 成本的 3 倍 | 用于监控金币是否失去意义 |

## 6. 建议 KPI

| KPI | 建议目标 | 用途 |
|-----|----------|------|
| `金币净流量 = faucet - sink` | 长时间接近 0，短时间允许正负波动 | 判断经济是否失衡 |
| `30/60/120 分钟金币库存分布` | 中位数不应持续高于下一个主 sink 成本的 3 倍 | 判断金币是否溢出 |
| `首次建成贸易站后到首次成功招募的时间` | 5-12 分钟 | 验证“资源转金币转成长”闭环是否成立 |
| `金币支出占比` | 招募 45%-60%，刷新 15%-25%，装备 20%-35% | 保证招募仍是主出口 |
| `自动交易利用率` | 可交易盈余中 60%-85% 被成功卖出 | 判断保留线和成交上限是否合理 |
| `连续 10 分钟无有效 sink 的玩家比例` | < 15% | 判断金币是否失去决策价值 |
| `发展度来源占比` | 研究/建筑/核心零件三大项均需长期有效 | 监控是否出现单项碾压 |
| `同发展度段 Boss 胜率` | 40%-70% | 验证发展度是否正确映射战备阶段 |

## 7. 为什么发展度公式需要重写

当前公式：

```text
development = clamp(
  key_research_count * 20 +
  key_building_level_sum * 8 +
  core_part_spent * 2 +
  active_combat_power / 50,
  0,
  100
)
```

存在以下问题：

- `key_building_level_sum * 8` 的理论上限已经超过 100，单靠建筑就能把发展度刷满，违背“三条长期成长线协同”的设计目标。
- `key_research_count` 只有 0-3 档，却每档跳 20 点，阶段跃迁过于离散，容易出现“研究一点就跨段”的硬跳变。
- `active_combat_power / 50` 直接吃原始战力，数值波动大，且会把装备与临时战备放大成发展度主导项，不符合“装备仅为辅助成长系统”的定位。
- `core_part_spent` 方向是对的，但当前只是线性叠加，缺乏与建筑、研究之间的协同约束。
- 公式是纯线性叠加，没有偏科惩罚或均衡奖励，与“只堆生产不做研究会停滞”的目标不一致。

## 8. 推荐的新公式

### 设计思路

- 保留 `研究`、`建筑`、`核心零件`、`战备` 四个维度。
- 把 `研究`、`建筑`、`核心零件` 作为主权重，`战备` 作为辅助权重。
- 所有输入先标准化到 `0-1`，再做加权，避免异构变量直接相加。
- 对战力使用 `sqrt` 压缩，避免战备或装备把发展度拉爆。
- 加入轻度偏科惩罚，鼓励三大主线同步推进，但不做硬锁。

### 推荐公式

```text
research_score = clamp(completed_key_research / 3, 0, 1)

building_score = clamp(
  0.15 * clamp(house_level / 3, 0, 1) +
  0.10 * clamp(kitchen_level / 2, 0, 1) +
  0.10 * clamp(farm_level / 3, 0, 1) +
  0.10 * clamp((lumbermill_level + quarry_level) / 6, 0, 1) +
  0.20 * clamp(research_bench_level / 3, 0, 1) +
  0.10 * clamp(warehouse_level / 2, 0, 1) +
  0.10 * clamp(recruit_station_level / 2, 0, 1) +
  0.10 * clamp(barracks_level / 2, 0, 1) +
  0.05 * clamp(trade_station_level / 2, 0, 1),
  0,
  1
)

core_score = clamp(core_part_spent / 12, 0, 1)

combat_score = clamp(sqrt(active_combat_power / target_combat_power), 0, 1)

base_development =
  100 * (
    0.35 * research_score +
    0.30 * building_score +
    0.20 * core_score +
    0.15 * combat_score
  )

balance_factor = 0.85 + 0.15 * min(research_score, building_score, core_score)

development = floor(clamp(base_development * balance_factor, 0, 100))
```

## 9. 变量范围与权重理由

| 变量 | 建议范围 | 权重 | 设计理由 |
|------|----------|------|----------|
| `research_score` | 0-1 | 0.35 | 研究决定科技层级与阶段推进，应是最强主线之一 |
| `building_score` | 0-1 | 0.30 | 建筑代表聚落成熟度，但不能靠单一建筑堆满 |
| `core_score` | 0-1 | 0.20 | Boss 突破必须有存在感，但不应高于研究和基础建设 |
| `combat_score` | 0-1 | 0.15 | 战备重要但应是辅助维度，且需压缩波动 |
| `target_combat_power` | 500-800 | - | 用于定义 V1 后段“合格战备线”，不宜过低 |
| `balance_factor` | 0.85-1.00 | - | 轻度惩罚偏科，防止只堆一条线就越级 |

### 为什么不把金币直接放进发展度

- 金币是中介货币，不是阶段本身。
- 把 `gold_stock` 或 `gold_income_per_min` 直接放进发展度，会让贸易站产币速度直接拉高 Boss 难度，破坏“金币服务招募与装备”的定位。
- 经济成熟度若需要体现，建议只通过 `trade_station_level` 进入 `building_score`，而不是把金币本身并入公式。

## 10. 边界条件

| 场景 | 预期行为 |
|------|----------|
| 玩家只堆建筑，不做研究 | `building_score` 可上升，但会被 `balance_factor` 压住，无法单线刷满发展度 |
| 玩家只打 Boss，不建设不研究 | `core_score` 上升有限，发展度不会因为囤零件或单线花零件而失控 |
| 玩家战力短时暴涨 | `combat_score` 通过 `sqrt` 压缩，最多只贡献 15% 主体分数 |
| 玩家金币暴增 | 不直接提高发展度，只提高招募/装备可行性 |
| 玩家暂停贸易站 | 金币 faucet 降低，但发展度不会瞬间下跌 |
| 玩家囤积核心零件不消费 | 仅 `core_part_spent` 计入发展度，不鼓励囤材料假进度 |
| 玩家没有贸易站 | 金币获取受限，但不应阻断前期基础建设与研究理解 |
| 食物接近保留线 | 停止卖粮，优先保证生存稳态 |
| 队列中有高成本建筑升级 | `pending_buffer` 全额预留，交易系统不得卖掉已计划消耗的关键资源 |
| 达到发展度上限 | 仍可继续积累金币、招募、装备，但不再通过单项刷数值继续推高阶段 |

## 11. 最终建议

- 金币系统在 V1 应收束为 `单主 faucet + 三主 sinks` 的结构：`贸易站 -> 招募 / 刷新 / 装备`。
- 发展度应从“原始变量线性相加”改为“标准化加权 + 轻度偏科惩罚”。
- 金币不应直接进入发展度公式，但贸易站等级可以作为建筑成熟度的一部分。
- 若装备系统暂未完成，必须优先把招募购买与刷新费用做稳，否则金币会在中期快速失去意义。
